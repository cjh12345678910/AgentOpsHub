# AgentOps Hub 缺口清单（OpenSpec Change 视角）

更新时间：2026-03-04
基线：基于当前代码实现（`backend`、`agent_runtime`、`frontend`）与项目描述文档对比。

## 现状结论

- 当前已具备：任务异步执行（线程池）、任务取消、trace 回放、RAG（txt/md）、工具治理基础页、PLAN/ACT/GENERATE/VERIFY/REPAIR runtime。
- 当前主要缺口：鉴权与持久化 RBAC、消息队列化调度、PDF 与 `file_write`、评估回归体系、可观测性、治理 UI 完整度、全链路 E2E 实证。

## Change 列表（建议按优先级执行）

## Change-01：`auth-and-rbac-hardening`

- 优先级：P0
- 目标：把当前“无鉴权 + 内存权限映射”升级为可落地的认证与 RBAC。
- 范围：
  - Backend 增加统一鉴权（JWT 或 session 二选一）。
  - 对 `/api/tasks/*`、`/api/tools/*`、`/api/docs/*`、`/api/rag/*` 增加认证/授权拦截。
  - `tools/permissions` 从内存 `ConcurrentHashMap` 改为数据库持久化。
  - 返回统一错误码：`UNAUTHORIZED`、`FORBIDDEN`。
  - 记录鉴权拒绝审计（actor/resource/reason）。
- 验收：
  - 未登录访问受保护接口返回 401 + `UNAUTHORIZED`。
  - 越权访问返回 403 + `FORBIDDEN`，并含 machine-readable reason。
  - 重启服务后权限配置不丢失。

## Change-02：`queue-worker-dispatch-with-dlq`

- 优先级：P0
- 目标：将当前应用内线程池异步升级为 MQ 驱动的任务调度与消费。
- 范围：
  - 引入 RabbitMQ（或 Kafka）用于 task dispatch。
  - `POST /api/tasks` 仅负责落库 + 入队。
  - Worker 消费并调用 `agent_runtime`，支持重试与 DLQ。
  - 取消任务与消息状态协同（运行中取消、队列中取消）。
- 验收：
  - 创建任务接口稳定快速返回（不阻塞 runtime 执行）。
  - 消费失败进入重试，超过阈值进入死信队列。
  - 取消后状态收敛正确，trace 可见取消节点。

## Change-03：`pdf-upload-and-file-write-tool`

- 优先级：P1
- 目标：补齐文档与产物输出能力。
- 范围：
  - Docs 支持 PDF 上传解析（保留 txt/md）。
  - 明确 PDF 解析失败语义与前端状态展示。
  - Runtime ToolRegistry 增加 `file_write` 工具。
  - `file_write` 增加受控目录、大小限制、扩展名限制和审计字段。
  - 任务详情页可展示文件产物信息（路径或下载入口）。
- 验收：
  - PDF 可进入 chunk/embedding/search 链路。
  - 非法路径写入（越界）被阻断并有 `denyReason`。
  - 产物可在任务详情可见。

## Change-04：`rag-grounded-generation-and-citation-fidelity`

- 优先级：P0
- 目标：让生成答案真正基于 RAG 证据内容，而不是只带 citation ID。
- 范围：
  - Runtime 在 ACT 阶段拿到 `chunk_fetch` 内容后，将证据片段注入 GENERATE prompt。
  - 生成阶段要求输出优先引用证据片段，减少纯模型先验回答。
  - Verifier 增加“答案-证据一致性/可追溯性”规则（后续完善）。
  - 前端可展示“答案所用证据片段摘要”（后续完善）。
- 验收：
  - 同一问题在有/无文档时，答案内容应明显受文档证据影响。
  - 回放中可见 chunk 内容被用于生成阶段（而非仅 chunkId）。
  - 引用与答案关键结论可对应核验。
- 当前状态：
  - 已完成第一阶段实现：`agent_runtime` 已在 GENERATE prompt 注入 `chunk_fetch` 证据文本（最多 5 条、每条截断），不再只传 citation ID。

## Change-05：`tool-governance-ui-completion`

- 优先级：P1
- 目标：把治理页面从“基础可看”补齐到“可运营”。
- 范围：
  - Tool Catalog：工具名搜索、状态/类型筛选。
  - Permission Config：前后端一致的冲突校验和错误提示。
  - Security Audit：增加时间范围过滤、单条详情视图（request/response 摘要、scope、rule）。
  - 后端 catalog 从硬编码迁移为注册中心读取（或配置驱动）。
- 验收：
  - 三个页面在真实数据下可稳定操作且无报错。
  - 审计页支持 `policyDecision + from + to` 组合过滤。
  - catalog 信息与 runtime 实际工具定义一致。

## Change-06：`offline-eval-regression-pipeline`

- 优先级：P1
- 目标：建立离线评估与回归防退化机制。
- 范围：
  - 定义 `eval_cases.jsonl`（20~50 条）。
  - 增加评估执行入口（API 或脚本均可）。
  - 产出指标：`task_success_rate`、`json_schema_pass_rate`、`citation_coverage_rate`、`repair_rate`、`avg/p95 latency`。
  - 保存批次结果并支持基线对比。
  - 前端增加评估批次与对比展示页（可先简版）。
- 验收：
  - 一键触发评估并产出 JSON 报告。
  - 可按 case 追溯失败原因。
  - 可比较“当前批次 vs 上次/基线”。

## Change-07：`observability-and-ops-baseline`

- 优先级：P2
- 目标：补齐企业化可观测与运行基线。
- 范围：
  - 接入 Prometheus metrics（任务量、失败率、阶段耗时、工具拒绝率）。
  - 基础 tracing（OpenTelemetry）串联 backend 与 runtime。
  - 结构化日志规范与 traceId 贯通。
  - 健康检查与关键告警阈值建议。
- 验收：
  - 能按 taskId/traceId 查询跨服务链路。
  - 有最小可用 dashboard（成功率、延迟、失败原因分布）。
  - 工具拒绝原因可按策略维度聚合。

## Change-08：`e2e-acceptance-evidence`

- 优先级：P0（归档前置）
- 目标：提供固定剧本的全链路验收证据，支撑归档和演示。
- 范围：
  - 固定输入与脚本覆盖：
    - 成功路径（PLAN->ACT->GENERATE->VERIFY）。
    - repair 路径。
    - scope deny 路径。
    - safety deny 路径（http/sql）。
    - cancel 路径。
  - 输出一次完整验收记录（命令、关键输出、截图或日志摘要）。
- 验收：
  - 任意同事可按文档复现同样结果。
  - 每条路径都有对应证据与期望结果。
